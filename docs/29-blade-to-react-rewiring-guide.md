# Blade-to-React Storefront Rewiring Guide

*Complete removal of Blade storefront links, URLs, redirects, and public-site assumptions for the old website, plus the correct frontend model for new customers.*

---

**Applies to**  
Old website migration where React frontend is being wired to the headless commerce backend.

**Core message**  
The old Laravel app still contains the Blade storefront inside the backend. All Blade-linked URLs, `route()` helpers, redirects, API-generated storefront links, and admin "view/open" actions must be rewired to the React storefront.

---

**Simple answer about the new website:**  
You will always have a frontend. The starter storefront is itself a frontend. For each new customer, you either customize that base frontend or create a new template/theme layer on top of the same backend APIs.

---

## 1. What is happening today

The old website is currently in a mixed state. The new React storefront exists and already calls backend APIs, but the backend Laravel app still behaves as if the public storefront is the old Blade website.

- The Blade storefront lives inside the backend Laravel application itself.
- Laravel web routes like `route('home')` and `route('product', slug)` still point to Blade pages.
- Admin "View Product", "Browse Website", and similar links still open the Blade storefront.
- Some API resources also generate storefront URLs using old Blade route helpers.
- A few update flows still follow old form-post patterns rather than clean API patterns.

## 2. Where the Blade storefront actually lives

In the old website, the Blade storefront is not a separate frontend service. It is part of the backend Laravel app.

| Area              | Typical location                  | Meaning                                                                                  |
|-------------------|-----------------------------------|------------------------------------------------------------------------------------------|
| Public routes     | `routes/web.php`                  | Home, product, category, shop, brand, and other Blade storefront routes                 |
| Public controller | `HomeController` / frontend controllers | Returns Blade views like `frontend.<theme>.index`                                  |
| Views             | `resources/views/frontend/*`      | Classic/Metro/Minima/Megamart/etc. Blade theme folders                                  |
| URL base          | `APP_URL`                         | Default Laravel base URL still used by old storefront helpers                           |

## 3. Why the old website keeps falling back to Blade

- Any `route()` helper that references a public storefront route still targets Laravel Blade pages.
- Any admin table, dashboard card, email, or resource class that constructs storefront links with `APP_URL` or `route('product')` still sends users to Blade.
- If React is built separately but not mounted as the public site, Laravel remains the default storefront.
- If APIs return product links generated by Blade routes, React receives legacy URLs and the old storefront leaks back into the user journey.

## 4. Known examples already identified

| Issue                   | Current behavior                                   | Problem                                         | Correct direction                                   |
|-------------------------|----------------------------------------------------|-------------------------------------------------|-----------------------------------------------------|
| View Product link       | `products_table.blade.php` uses `route('product', slug)` | Always opens old Blade PDP                      | Use `storefront_url('product/<slug>')`              |
| Browse Website          | Admin opens `route('home')`                        | Always opens Blade homepage                     | Use configurable `FRONTEND_URL`                     |
| Product update mutation | React uses `_method=PUT` form spoofing             | API middleware group relies on web-form spoof flow | Call real API route directly with `api.put()` or align to a POST update route intentionally |
| API link fields         | Some resources generate product/shop URLs with `route()` | React receives Blade URLs                       | Return storefront-aware URLs or relative paths only |

## 5. Rewiring goal

The old website must move to this final model:

- React storefront becomes the public customer website.
- Laravel remains the backend, API server, admin panel, and optional seller/backend management layer.
- No customer-facing URL should be generated from Blade storefront helpers anymore.
- All public links should resolve from a storefront-aware URL strategy.
- The old Blade storefront may remain only as a temporary fallback during migration, then be retired from public use.

## 6. Complete rewiring scope

To finish the migration, remove Blade-linked URL generation and Blade-oriented public assumptions from every one of the following areas.

### A. Backend admin views

- View Product / Open Product / Visit Product links
- Browse Website button
- Open Category / Brand / Shop links
- Dashboard cards and reports that link to public pages

### B. API resources and serializers

- Product link fields
- Category, brand, and seller public URLs
- Share URLs
- Any URL composed with `route()` or `APP_URL`

### C. Redirects and public responses

- Login/register success redirects that point to web storefront routes
- Payment success/failure/cancel redirects
- Order confirmation redirects

### D. Email and notification links

- Order emails
- Password reset or account activation links if storefront-facing
- Promotional emails linking to product/category pages

### E. Public web routing and deployment

- Public site root must be React, not Laravel Blade
- `/api` remains Laravel
- `/admin` remains Laravel
- Fallback routing must go to `React` `index.html` for storefront paths

## 7. Mandatory backend changes

These changes should be implemented in the old website backend before the final cutover.

- Add `FRONTEND_URL` to environment config.
- Add a `config/storefront.php` file with a single source of truth for the public storefront base URL.
- Create a `storefront_url()` helper that generates public URLs using `FRONTEND_URL` instead of `route('home')` or `APP_URL`.
- Replace all public-facing `route()` helpers in backend views, resources, emails, and redirect logic with `storefront_url()` or relative public paths.
- Decide one consistent product update pattern: either a real API `PUT/PATCH` route or an intentional `POST` update endpoint for admin mutation. Do not mix web-form spoofing assumptions with API-only routes.

## 8. Recommended helper pattern

Use a dedicated storefront helper. The exact implementation can vary, but the behavior should be consistent.

Example:

```env
FRONTEND_URL=https://your-public-storefront.com
```

```php
config('storefront.url') => env('FRONTEND_URL', env('APP_URL'));

storefront_url('product/' . $slug);
```


## 9. Frontend React rewiring requirements

The React storefront should become the single customer-facing frontend for the old website migration. The following must be checked module by module.


| Module | Backend must expose | React must consume/render | Status target |
| :-- | :-- | :-- | :-- |
| Site config | branding, logos, currency, settings, links | global app config, header, footer, SEO defaults | Fully wired |
| Navigation | menu/footer content | header, mobile nav, footer | Fully wired |
| Homepage | banners, hero, featured sections, campaign blocks | home page composition and ordering | Fully wired |
| Catalog | categories, brands, filters, products, stock | listing pages, PDP, search, filters | Fully wired |
| Cart | cart CRUD, totals, coupon rules | add/update/remove, totals, mini-cart, cart page | Fully wired |
| Checkout | shipping, address, payment-init, order create | checkout steps, success/failure handling | Fully wired |
| Account | auth, profile, addresses, orders, wishlist | login/account pages and protected routes | Fully wired |
| CMS | pages, blog, policies, contact content | content pages and SEO routing | Fully wired |

## 10. Public storefront activation for the old website

Once rewiring is complete, activate React as the live public storefront.

- Public site URL should point to the React storefront deployment.
- Admin "Browse Website" should open `FRONTEND_URL`, not `route('home')`.
- Web server or proxy should send public storefront paths to React.
- `/api` should continue to route to Laravel.
- `/admin` should continue to route to Laravel admin.
- Legacy Blade storefront should remain only as a fallback until cutover is proven stable.


## 11. Migration order for the old website

1. Introduce `FRONTEND_URL` and `storefront_url()`.
2. Replace all backend admin "view/open/browse" links.
3. Replace API resource URLs and share links that still use Blade routes.
4. Review payment redirects, emails, and public notification links.
5. Standardize admin product update calls so API mutation does not depend on old form spoofing behavior.
6. Complete module-by-module React wiring for catalog, homepage, content, cart, checkout, and account.
7. Deploy React as the public site and switch the admin button to the storefront URL.
8. Keep Blade only as rollback fallback, then retire it from public use.

## 12. Acceptance checklist

- No admin "view/open/browse" action opens a Blade storefront page.
- No customer-facing API payload contains a Blade storefront URL.
- Public product, category, brand, seller, cart, account, and content links open in React.
- Checkout, payment success, payment failure, and order confirmation all return users to React routes.
- Emails and public notifications link to React storefront URLs where applicable.
- The public domain resolves to React storefront pages, while `/api` and `/admin` remain on Laravel.
